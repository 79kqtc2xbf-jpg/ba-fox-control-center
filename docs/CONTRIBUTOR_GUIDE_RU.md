# MF Group Tracker — как дорабатывать код

## 1. Какой доступ нужен

Для тестирования дашборда GitHub не нужен.

Для изменения кода каждому разработчику нужны:

- собственный GitHub-аккаунт;
- доступ к репозиторию `79kqtc2xbf-jpg/ba-fox-control-center`;
- локальная копия проекта;
- отдельная рабочая ветка.

Нельзя передавать коллегам личный GitHub-токен, пароль Google, содержимое `.env`, сервисные аккаунты или Apps Script credentials. Если человеку нужен доступ, его добавляют в GitHub как отдельного участника.

## 2. Безопасный процесс изменений

1. Обновить локальную ветку `main`.
2. Создать отдельную ветку с понятным названием.
3. Внести небольшое, связанное изменение.
4. Проверить его локально.
5. Создать Pull Request в `main`.
6. Дождаться автоматических проверок и просмотра изменений.
7. Сливать PR только после согласования.

Напрямую изменять `main` и обходить Pull Request не следует.

## 3. Где находится код

- `web/` — интерфейс дашборда;
- `apps-script/` — серверные маршруты, доступ к Google Sheets и проверка профиля;
- `tests/` — автоматические проверки;
- `bot/` и `services/` — Telegram-бот и Python-сервисы;
- `docs/` — инструкции, архитектура и история этапов.

Google Sheets остаётся рабочим источником данных. GitHub — источник правды для кода и документации.

## 4. Локальная проверка интерфейса

Запустите статический сервер из корня проекта:

```bash
python3 -m http.server 8000 --directory web
```

Откройте:

```text
http://localhost:8000
```

Не добавляйте рабочие endpoint, токены или Google Client ID в коммит. Локальные настройки должны оставаться локальными.

## 5. Проверки перед Pull Request

```bash
node --test tests/*.test.cjs
node --check web/app.js
node --check web/api/baFoxClient.js
python -m compileall bot services
git diff --check
```

Также проверьте интерфейс на широком и мобильном экране.

## 6. Правила для Apps Script

Изменения в `apps-script/` сначала проходят Pull Request и локальные тесты. Самостоятельно развёртывать Apps Script, менять Script Properties, OAuth, владельца проекта или URL веб-приложения нельзя без согласования с владельцем продукта.

После одобренного изменения назначенный сопровождающий:

1. синхронизирует исходники с Apps Script;
2. создаёт новую версию существующего веб-развёртывания;
3. сохраняет прежний URL `/exec`;
4. проверяет закрытый доступ без Google-токена;
5. проверяет вход тестового пользователя.

## 7. Что написать в Pull Request

- что изменено;
- зачем это нужно пользователю;
- какие данные и роли затрагиваются;
- какие тесты выполнены;
- нужны ли миграция Google Sheets или Apps Script deployment;
- как откатить изменение;
- скриншоты для заметных изменений интерфейса.

## 8. Когда остановиться и запросить согласование

Не продолжайте самостоятельно, если изменение требует:

- доступа к реальным задачам или персональным данным;
- нового секрета, токена или OAuth-разрешения;
- изменения структуры Google Sheets;
- изменения ролей или списка пользователей;
- удаления данных;
- развёртывания Apps Script;
- прямого изменения `main`;
- подключения платного сервиса.

В этих случаях сначала опишите изменение, риск и план отката владельцу продукта.
