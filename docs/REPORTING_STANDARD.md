# BA Fox — Reporting Standard

## Purpose

This document fixes the approved reporting style for Lisa's BA / Executive Assistant work.

The goal is to make daily and weekly reports look like ownership and process management, not like a loose list of actions.

## Core principle

Do not report tasks as:

```text
I did / pinged / waiting / in progress
```

Report them as:

```text
Result → Current status → Next action → Deadline / escalation point
```

## Main framing

The reporting goal is to show:

- what is already closed or nearly closed;
- what is actively managed;
- what is blocked and how the blocker is being handled;
- what the next clear focus is.

## Daily report structure

Use four blocks:

```text
✅ DONE / Finalization
🔄 In progress / Under control
⚠️ Blockers / Managed risks
🎯 Focus
```

A fifth block is allowed when needed:

```text
⏳ Wait list / Fixed external dependency
```

Use `Wait list` only for items where Lisa does not currently own the next operational action, or the item is explicitly parked as a known external dependency.

### 1. DONE / Finalization

Use this block first to reduce tension and show visible results.

Format:

```text
• [Task / organization] — [specific result / completion percentage / sent / prepared / transferred]
```

Examples:

```text
• Bitazza — document package is 70% complete; finalizing and sending today.
• JDB — documents submitted; currently under review.
• Kasikorn — required document package prepared.
• Presentations — finalized and sent to partners.
```

### 2. In progress / Under control

This block is for tasks that are not done yet but are actively managed.

Do not say only `waiting`.

Format:

```text
• [Task / organization] — current status → next action → deadline / escalation condition
```

Examples:

```text
• BCEL — follow-up sent on PromptPay; waiting for a reply until [date], then escalating.
• GLN — no stable reply; restoring communication through alternative contacts.
• P3 Estates — pushing for a specific meeting slot next week.
• Shobana — no response; switching to other entry points.
```

### 3. Blockers / Managed risks

This block is for things that are stuck because of external dependency, payment issue, document dependency, technical issue, or third-party response.

The framing must be:

```text
blocker → action taken → next attempt / deadline
```

Examples:

```text
• Laos office payment — payment is blocked; testing an alternative payment method today.
• US fee — payment does not go through; contacted support and checking an alternative payment route.
• BOL — waiting for translation; next stage is prepared and will start immediately after translation is received.
```

### 4. Focus

End with 1–3 priorities.

Recommended closing sentence:

```text
Main focus now is to close accumulated tails and move tasks into final statuses so they do not keep rolling over.
```

Russian version:

```text
Основной фокус сейчас — закрыть накопленные хвосты и перевести задачи в финальный статус, чтобы не тянуть их дальше.
```

## Forbidden weak formulations

Avoid these as standalone status updates:

```text
жду
в процессе
попробую
пинговала
делала
занималась
нет ответа
```

They are allowed only if followed by next action and deadline.

Better replacements:

```text
жду → ожидаю до [date], затем эскалирую / иду через другой канал
в процессе → сейчас на этапе [stage], следующий шаг [action] до [date]
попробую → сегодня проверяю [method], если не сработает — [backup]
пинговала → follow-up отправлен; следующий контроль [date]
нет ответа → ответа нет; перехожу к альтернативному контакту / каналу
```

## Owner-position rule

For every task ask:

```text
What does DONE look like?
```

If DONE is unclear, define it before reporting the task.

Examples:

```text
Bitazza DONE = full corrected KYB package sent and acknowledged.
BCEL DONE = final reply received or escalation route chosen.
P3 DONE = meeting slot confirmed or alternative decision maker contacted.
```

## BA Fox implementation requirement

When BA Fox prepares reports, it should:

1. Group tasks into DONE / In progress / Blockers / Focus.
2. Rewrite weak statuses into owner-style statuses.
3. Add next action and deadline where possible.
4. Avoid presenting `waiting` as a passive state.
5. Show controlled blockers as managed risks, not failures.
6. Keep the tone calm, direct, operational, and confident.
7. Preserve Lisa's real wording when it contains important operational facts.
8. Do not invent completion if Lisa only provided a blocker or a dependency.

## Notes from the approved example

- If a blocker is caused by internal resource limits, say it plainly but operationally: `нет денег на корпоративной карте → перенос контроля на понедельник`.
- If a thread is closed, state it as a management decision: `взаимодействие закрыто`, not as an endless `нет ответа`.
- If a vendor does not provide a requested product, capture that as a result: `виртуальная карта: у них её нет`.
- `Wait list` is acceptable when the item is a fixed external dependency and does not need to be presented as active work.
