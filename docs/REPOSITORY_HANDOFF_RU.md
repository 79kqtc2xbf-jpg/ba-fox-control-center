# BA Fox / MF Group Tracker — быстрый вход в репозиторий

> Назначение: этот файл — стартовая точка для нового разработчика или ИИ-агента. Он описывает актуальное устройство проекта, границы безопасности и порядок работы. Не считайте старые stage-документы текущим планом без сверки с кодом и последними коммитами.

## 1. Что это за продукт

BA Fox — внутренний трекер задач и рабочая панель для MF Group. Он помогает создавать и контролировать задачи, видеть сроки и напоминания, вести проекты и сохранять аудит изменений.

Основные пользователи — сотрудники домена `mfstream.io`. Владелец продукта и архитектуры процесса — Лиза Киселёва.

## 2. Актуальная архитектура

```text
Пользователь в браузере
        │ Google Sign-In / ID token
        ▼
GitHub Pages: web/ (HTML, CSS, JS)
        │ GET/POST к Apps Script Web App
        ▼
Apps Script: apps-script/
  ├─ проверка Google-профиля, роли и видимости
  ├─ логика задач и проектов
  ├─ журнал аудита
  └─ безопасные read/write маршруты
        │
        ▼
Google Sheets: Tasks, Users, Projects, AuditLog,
               Reports, NotificationQueue, Contacts
```

Дополнительный, не основной контур:

```text
bot/ + services/ → legacy Telegram/Railway MVP
```

Его сохраняют для совместимости и справки; новый функционал по умолчанию относится к `web/` и `apps-script/`.

### Границы ответственности

| Слой | Где искать | За что отвечает |
| --- | --- | --- |
| Интерфейс | `web/` | Экран задач, проектов, фильтры, формы, Google Sign-In, состояние UI |
| HTTP-клиент | `web/api/` | Конфигурация, запросы к Web App, mock fallback, локальное UI-состояние |
| Backend | `apps-script/` | Маршрутизация, авторизация, доступ к таблице, бизнес-правила, аудит |
| Данные | Google Sheet | Операционные задачи, пользователи, проекты, история аудита |
| Старый MVP | `bot/`, `services/` | Telegram-интерфейс и Python-сервисы; не менять попутно |
| Проверки | `tests/` | Контракт безопасной авторизации, аудит и проекты |

## 3. Ключевые правила безопасности

1. Не коммитить секреты: URL Web App, Sheet ID, OAuth client ID, action token, webhook, `.env`, service-account JSON и реальные данные задач.
2. Публичный frontend не должен получать `BA_FOX_ACTION_TOKEN`. Запись разрешается только после проверки Google ID token на backend и при `SAFE_WRITE_MODE=true`.
3. Доступ ограничен доменом `mfstream.io`, активным пользователем в листе `Users`, его ролью и настройками видимости.
4. Изменение схемы Google Sheet, ролей, списка пользователей, конфигурации Apps Script, развёртывания или реальных данных требует явного согласования владельца продукта и понятного отката.
5. Сначала код и PR, затем ручное развёртывание Apps Script назначенным сопровождающим. Не размещать URL или секреты в PR, документации и скриншотах.

## 5. Как ориентироваться в коде

### Web

`web/index.html` подключает интерфейс. Основная экранная логика находится в `web/app.js`; стили — в `web/styles.css`.

Клиентский слой не должен обращаться к Google Sheets напрямую. Используйте `web/api/baFoxClient.js` и существующий контракт Apps Script. В обычном репозитории включён mock fallback: локальный runtime-файл с endpoint не коммитится.

### Apps Script

Начните с `apps-script/Config.gs`, `WebApp.gs` и `WebApi.gs`.

- `WebApp.gs` — `doGet`/`doPost` и разбор маршрута;
- `TaskService.gs` — задачи, статусы, напоминания, архивирование;
- `ProjectsService.gs` — проекты;
- `IdentityService.gs` — проверка Google-профиля, роль и видимость;
- `SheetsStore.gs` — чтение/запись в Sheets;
- `AuditLog.gs` — аудит;
- `Validation.gs` и `DateTime.gs` — общие правила.

`Config.gs` содержит только безопасные значения по умолчанию. Фактические секреты и runtime-настройки живут в Apps Script Properties, не в Git.

### Состояние и совместимость

Репозиторий развивался по этапам, поэтому часть `STAGE_*` документов описывает прежний read-only или Telegram-first период. При противоречии приоритет такой:

1. действующий код и автоматические тесты;
2. этот handoff и последние коммиты;
3. свежий специализированный документ;
4. исторические stage-документы.

Перед изменением авторизации, прав или схемы обязательно запросите подтверждение, даже если старый документ разрешает действие.

## 6. Проверка перед передачей

Для изменения интерфейса и backend-контракта:

```bash
node --test tests/*.test.cjs
node --check web/app.js
node --check web/api/baFoxClient.js
git diff --check
```

Если менялись Python-файлы legacy-бота:

```bash
python -m compileall bot services
```

Для UI вручную проверьте широкий и мобильный экран. Не заявляйте об успешном Apps Script deployment, доступе к реальным данным или живом QA, если этого не было в рамках задачи.
